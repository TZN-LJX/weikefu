## 2. 产品概述

本产品将《威科夫操盘法》原书及个人核心笔记转化为一个可在手机上每日持续使用的学习、识图、解释、历史回放和模拟交易系统。

产品不是交易信号工具。它要求用户先独立判断市场背景和价量证据，再通过规则或 AI 获得反馈，从而建立统一、可复盘、受风险约束的 ETH 合约交易流程。

产品采用“公开应用外壳 + 本地私有学习包”：GitHub Pages 只托管界面和功能，完整 PDF、原书图片、原文、课程和私有案例通过 `.wkf` 文件在手机本地导入，不随公开网站发布。

产品采用“自适应闯关主线 + 每模块隐藏未来 ETH 回放”的学习方式，目标是在约 90 天内帮助用户完成从概念记忆到陌生图表分析和合规模拟交易的迁移。

## 3. 用户与使用场景

### 3.1 目标用户

第一版只服务于一名私人用户：

- 已阅读过一遍《威科夫操盘法》。
- 知道部分术语，但尚不能稳定识别市场背景。
- 有 Binance ETHUSDT 合约交易经验，当前总体亏损。
- 常用 5 至 10 倍杠杆。
- 主要问题是背景判断不清、仓位偏大、缺少统一系统。
- 每天可持续投入约 15 至 20 分钟学习。

### 3.2 核心使用场景

1. 每天在手机上完成系统生成的学习任务。
2. 首次使用时从手机文件中导入私人学习包。
3. 在基于原书概念的图解中识别供需、阶段和威科夫事件。
4. 在隐藏未来走势的 ETH 历史片段中完成多周期判断。
5. 用固定结构解释自己的分析逻辑。
6. 在风险规则约束下完成内部模拟交易。
7. 复盘交易结果，区分分析、执行、风险和情绪错误。
8. 第二阶段在实时行情中先自行判断，再让系统核对。

## 4. 用户需要完成的工作

产品帮助用户完成以下 Job To Be Done：

- 当我每天只有少量时间时，我需要系统告诉我今天最该学什么，以便持续推进而不是反复阅读。
- 当我面对陌生 ETH 图表时，我需要按固定顺序判断背景和证据，以免凭单个形态冲动交易。
- 当我判断错误时，我需要知道遗漏了哪些证据并重新解释，以免只记住答案。
- 当我准备模拟下单时，我需要强制检查风险和失效条件，以免仓位超过承受能力。
- 当一笔交易亏损时，我需要区分决策质量和交易结果，以免用单次盈亏否定或强化错误方法。

## 5. 产品目标与成功指标

### 5.1 90 天核心目标

| 指标 | 目标 |
| --- | --- |
| 持续学习 | 连续学习至少 30 天 |
| 基础知识 | 正确率达到 80% 以上 |
| 陌生图表判断 | 正确率达到 90% 以上 |
| 解释能力 | 能完整说明背景、证据、行动和失效条件 |
| 模拟实践 | 完成至少 30 笔遵守规则的内部模拟交易 |

### 5.2 辅助指标

- 今日任务完成率。
- 延迟复测通过率。
- “不交易”判断的合理率。
- 模拟交易规则遵守率。
- 仓位、止损和分析顺序违规次数。
- 模拟交易 R 倍数分布和最大回撤。

实盘盈利只作为长期观察指标，不作为收益承诺或课程解锁条件。

## 6. 产品原则

1. 原书优先：原书定义高于笔记和 AI 解释。
2. 用户先判断：系统和 AI 不在用户提交前给出结论。
3. 背景优先：任何局部形态必须放在 4 小时背景和 1 小时结构中判断。
4. 证据优先：评分关注价量证据是否支持结论，不只匹配形态名称。
5. 风险优先：模拟交易必须满足单笔风险不超过账户权益 1%。
6. 允许不交易：不是每段行情都需要产生订单。
7. 过程与结果分离：合格决策可能亏损，错误决策也可能暂时盈利。
8. 无 AI 可用：AI 故障不得阻断核心学习流程。
9. 单用户低成本：第一版不建设账号、付费服务和长期业务后端。

## 9. 核心用户流程

### 9.1 每日学习流程

`打开首页 -> 查看今日任务 -> 复习错题 -> 学习新知识 -> 标注原创教学图解 -> 完成 ETH 回放 -> 费曼解释 -> 更新掌握度`

### 9.2 历史回放流程

`查看 4h 背景 -> 判断 1h 阶段 -> 比较价量波动 -> 标记关键位置和事件 -> 判断是否交易 -> 给出计划与失效条件 -> 提交 -> 反馈 -> 揭示未来走势`

### 9.3 模拟交易流程

`完成交易前检查 -> 输入入场/止损/目标/杠杆 -> 计算风险与仓位 -> 通过或阻止 -> 推演历史走势 -> 结束交易 -> 填写复盘`

### 9.4 错误反馈流程

`提交错误判断 -> 标出遗漏证据 -> 苏格拉底式反问 -> 用户重新解释 -> 解释合格 -> 显示参考分析 -> 揭示走势 -> 加入间隔复习`

### 9.5 首次导入流程

`打开公开网站 -> 配置或跳过 AI -> 选择 .wkf 文件 -> 本地校验 -> 检查存储空间 -> 用户确认 -> 写入浏览器数据库 -> 进入今日任务`

## 11. 核心业务规则

### 11.1 内容规则

- 原书结论与核心笔记冲突时，以原书为准。
- ETH 适配不得改变原书核心逻辑。
- 每个正式案例必须包含来源、可见区间、隐藏区间、标准证据和人工审核状态。

### 11.2 解锁规则

- 单元正确率达到 90%。
- 固定解释结构已经填写完整。
- 必填证据与结论不存在直接冲突。

### 11.3 风险规则

- 风险金额 = 当前模拟账户权益 x 1%。
- 止损距离比例 = abs(入场价 - 止损价) / 入场价。
- 风险约束仓位 = 风险金额 / 止损距离比例。
- 保证金约束仓位 = 可用模拟权益 x 杠杆倍数。
- 最大名义仓位 = min(风险约束仓位, 保证金约束仓位)。
- 计划 R 倍数 = abs(目标价 - 入场价) / abs(入场价 - 止损价)。

### 11.4 AI 规则

- 确定性评分先执行，AI 反馈后执行。
- AI 结果只影响解释建议，不得修改风险计算。
- AI 返回格式异常时，不更新掌握度。

## 13. 非功能需求

### 13.1 兼容性

- 主要验收设备：华为 Mate 80。
- 主要验收系统：HarmonyOS 6.1.0。
- 主要验收浏览器：华为浏览器 6.1.6.305。
- Android Chrome。
- iPhone Safari。
- 桌面 Chrome 和 Edge。
- 手机最小宽度 360px。

### 13.2 可用性

- 普通学习流程适合手机纵屏和单手操作。
- 精细标注支持横屏全屏。
- 文字、按钮、图表和反馈不得重叠或裁切。
- 用户输入在请求失败后不得丢失。

### 13.3 性能

- 今日首页优先加载本地任务和进度。
- 课程、图片和图表数据按需加载。
- AI 和行情请求不得阻塞已经载入的核心学习界面。
- 网络中断时必须提示重新联网，不承诺离线继续学习。

### 13.4 安全与隐私

- 不收集 Binance 账户信息。
- 不保存 Binance API Key。
- AI Key 只保存在用户浏览器中。
- AI Key 不进入日志或备份。
- 学习数据默认只保存在用户浏览器中，不上传到 GitHub。
- 网站和仓库公开可访问，因此不得发布原始 PDF、原书原图、大段原文或任何密钥。
- 私有学习包仅由浏览器本地读取和保存；除当前题目所需的最小 AI 上下文外，不得上传包内内容。

## 14. 异常处理

| 场景 | 产品行为 |
| --- | --- |
| 网站无法联网 | 显示需要联网的明确提示，不承诺离线启动或离线课程 |
| AI 无 Key、超时或余额不足 | 保留输入，使用固定规则反馈，允许继续学习 |
| 行情数据中断或陈旧 | 明确显示状态，停止实时评分，不伪装为实时数据 |
| 本地保存失败 | 明确提示保存失败并建议导出备份 |
| 导入文件损坏 | 拒绝覆盖当前数据，显示原因 |
| 案例缺少来源或隐藏区间 | 不进入正式题库 |
| 私有学习包格式或校验失败 | 拒绝导入，不覆盖已有内容 |
| 浏览器存储空间不足 | 在写入前停止并显示所需空间 |
| 私有学习包导入中断 | 清理未完成数据，不产生半导入状态 |
| 评分规则异常 | 保留作答，将该题标记为不可计分 |
| 风险计算无效 | 阻止模拟订单并逐项显示错误 |
| 重复提交订单 | 不创建重复交易记录 |

## 15. 第一版验收标准

### 15.1 内容验收

- 原书五章核心方法均有课程映射。
- 每个知识单元都有来源、图解、练习和纠错方式。
- 原书内容和 ETH 适配内容标识清楚。
- 正式课程与案例完成人工审核。

### 15.2 学习验收

- 首页能够生成今日任务。
- 错题能够进入间隔复习并完成延迟复测。
- 解锁同时检查正确率和解释完成度。
- 没有 AI Key 时仍可完成核心学习流程。

### 15.3 图表验收

- 提交答案前无法查看未来 K 线。
- 4h 与 1h 案例时间对齐正确。
- 手机纵屏分步训练和横屏标注均可完成。
- 证据充分的多种合理答案可被接受。

### 15.4 模拟交易验收

- 风险金额、最大仓位和 R 倍数计算正确。
- 超过 1% 风险的订单无法提交。
- 做多与做空止损方向错误时无法提交。
- 合格亏损可记录为“过程正确、结果亏损”。

### 15.5 数据验收

- 导出数据能够在新浏览器中完整恢复。
- API Key 不出现在备份中。
- 数据失败时不会显示虚假的成功或实时状态。

### 15.6 响应式验收

- 360px 手机宽度下无关键内容重叠、裁切或无法点击。
- 华为 Mate 80、HarmonyOS 6.1.0、华为浏览器 6.1.6.305 能通过网址完成核心流程并添加到桌面。
- Android Chrome、iPhone Safari、桌面 Chrome / Edge 可完成核心流程。

### 15.7 私有学习包验收

- 华为浏览器能够从手机文件中导入使用真实两份 PDF 生成的 `.wkf` 包。
- 导入期间没有包内容上传到网络。
- 导入后能够查看完整 PDF、原书图片、原文、课程和私有案例。
- GitHub Pages 更新或页面刷新后，本地学习包仍然存在。
- 损坏文件、版本不兼容、空间不足和中断导入不会破坏已有内容或进度。
- 普通进度备份不包含包内出版物内容。

## 16. 依赖与风险

### 16.1 内容制作工作量

原创教学图解制作、ETH 案例筛选和标准证据标注是第一版最大的工作量。产品开发可以先使用少量经过审核的代表性内容打通完整流程，再按课程映射表逐步补齐。

### 16.2 图表答案存在歧义

威科夫分析不是所有场景都有唯一标签。题库必须保存可接受结论、必要证据和反证条件，不能只保存一个字符串答案。

### 16.3 AI 浏览器直连限制

部分 AI 服务商不允许浏览器直接跨域调用。第一版不建设付费代理；选择 AI 服务商时必须验证浏览器 CORS 支持，否则 AI 功能暂时关闭。

### 16.4 本地数据丢失

免登录模式下，删除浏览器数据会丢失进度。导出备份必须在第一版提供，并在关键节点提醒用户备份。

### 16.5 盈利预期偏差

产品必须持续强调训练的是决策质量和风险纪律，不把短期胜率或盈利包装为产品保证。

### 16.6 公开静态部署风险

GitHub Pages 和公开仓库不存在真正的访问控制。`noindex`、`robots.txt` 和不易猜测的网址只能降低被发现概率，不能阻止访问或下载。公开构建只能包含个人摘要、必要短引用和原创图解，原始 PDF、原书原图、大段原文、API Key 和用户备份必须排除。

### 16.7 华为浏览器兼容性

华为浏览器对标准 PWA 安装提示的支持可能与 Chrome 不同，因此第一版验收目标是通过 HTTPS 网址完成核心流程并添加到桌面，不要求离线课程。兼容性必须在华为 Mate 80 真机上验证，不能只依赖桌面模拟。

### 16.8 私有学习包存储风险

私有包会占用华为浏览器本地存储，实际容量取决于浏览器策略。开发时必须用包含两份真实 PDF 和派生资源的完整包测试容量、持久化和更新行为。清除浏览器数据会删除已经导入的包，用户需要保留原始 `.wkf` 文件以便重新导入。

### 16.9 私有学习包传播风险

第一版 `.wkf` 文件不做密码加密。任何获得该文件的人都可能读取其中内容，因此文件只能通过 USB、华为分享或私人网盘在用户自己的设备之间传输，不得上传到公开仓库或公开链接。

## 17. 开发前置准备

### 17.1 启动开发前必须完成

1. 用户最终批准本 PRD。
2. 初始化 Git 仓库并建立首次文档基线提交。
3. 形成原书章节到课程阶段的内容映射表。
4. 选出第一批需要转化为原创教学图解的原书概念和 ETH 历史案例。
5. 确认第一版使用的 AI 兼容接口，并验证浏览器 CORS。
6. 使用华为 Mate 80、HarmonyOS 6.1.0、华为浏览器 6.1.6.305 作为主要手机验收设备。
7. 在实施计划中确定前端框架、图表库、本地数据库和测试工具。
8. 将本地项目连接到公开仓库 `TZN-LJX/weikefu`，并完成首次 Git 推送授权。
9. 在本机生成第一版 `.wkf` 私有学习包，并保留一份不会进入 Git 的源文件。

### 17.2 可以与开发并行准备

1. 按统一格式整理知识单元、术语表和来源页码。
2. 根据原书概念制作原创教学图解并建立素材命名规则。
3. 从 ETHUSDT 历史数据中筛选典型、模糊、失败和不交易案例。
4. 为案例标记 4h 背景、1h 阶段、关键证据、可接受答案和失效条件。
5. 准备用户真实交易中的匿名复盘样本，用于验证日志字段是否够用。
6. 配置 GitHub Pages 和自动部署，并验证 `/weikefu/` 基础路径。
7. 建立仓库排除规则，确保原始 PDF、`APIKEY.txt`、环境变量文件和用户备份不会被提交。
8. 使用 USB、华为分享或私人网盘把 `.wkf` 文件传到华为手机进行导入验收。

### 17.3 第二阶段前再准备

1. Binance 公开历史和实时数据接口的限频与断线策略。
2. 实时 K 线收盘判定和本地时区转换测试数据。
3. 浏览器语音识别或第三方语音转写服务。
4. 实时模式的陈旧数据、重连和缺失 K 线处理方案。
